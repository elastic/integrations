# HarfangLab EDR Integration for Elastic

## Overview

The HarfangLab EDR integration collects security telemetry from [HarfangLab](https://www.harfanglab.io/) (Hurukai) Endpoint Detection and Response. It supports two collection methods that can be enabled independently or in parallel:

- **REST API** — Elastic Agent polls the HarfangLab Manager REST API for **alerts**, **threats**, and **agent inventory** using a static API token.
- **Syslog** — Elastic Agent listens for RFC 5424 messages with a JSON body forwarded by the HarfangLab Syslog connector, covering 14 vendor `log_type` values (alerts, threats, agent events, process, network, file, driverload, eventlog, auditlog, connectionlog, injectedthread, investigation, fallback for unknown types).

Events are normalized to the Elastic Common Schema (ECS) by the integration's ingest pipelines, with vendor-specific fields preserved under `harfang_labs.*` for forensic context.

### Compatibility

This integration has been tested against the HarfangLab REST API as exposed by Manager versions `9.x` (API path prefix `/api/data/alert/alert/` and `/api/data/endpoint/`) and the HarfangLab Syslog connector emitting RFC 5424 with a JSON payload.

It is compatible with Elastic stack `8.19+` and `9.1+` (see manifest `conditions.kibana`).

### How it works

For the **API** policy template, the `cel` input authenticates with a static `Token` header and walks the standard Django REST Framework paginated endpoints (`?offset=...&limit=...&ordering=...`). Each data stream persists a `last_seen` cursor (latest `alert_time` / `creation_date` / `lastseen` in the returned page) and resumes from it on the next poll. Three streams run in parallel with independent intervals: `alert` (1m default), `threat` (1m), and `agent` (5m).

For the **Syslog** policy template, the `tcp` and `udp` inputs receive raw RFC 5424 lines, the agent's `syslog` processor extracts the header into `log.syslog.*` and exposes the JSON body in `message`. The default ingest pipeline parses the JSON, reads `log_type`, and routes events to a per-type sub-pipeline that performs the ECS mapping.

## What data does this integration collect?

The HarfangLab EDR integration collects events of the following types:

- **Alerts** (REST API `alert` and Syslog `log_type: alert`): YARA, Sigma, HLAI, IOC, ransomware, antivirus and other detections including alert metadata, agent context, process / file / network indicators and matched rules.
- **Threats** (REST API `threat` and Syslog `log_type: threat`): correlated incidents aggregating related alerts, impacted agents and users, MITRE ATT&CK tactics, and the top contributing rules.
- **Agent inventory** (REST API `agent` and Syslog `log_type: agent`/`agentlog`): EDR agent state (status, last_seen, OS, IP) for asset visibility and posture monitoring.
- **Endpoint telemetry** (Syslog only): process events, network connections, file operations, driver loads, Windows event logs, audit logs, injected thread detections, investigation actions, and uncategorized telemetry routed through a fallback pipeline.

### Supported use cases

- Security operations: triage HarfangLab alerts in Elastic Security with full ECS categorization and correlation against host, user, and process context.
- Threat hunting: search vendor-specific endpoint telemetry (process trees, hash matches, signature data) using ES|QL or Discover.
- Asset visibility: track HarfangLab-managed endpoints, their last check-in time, OS, and connectivity status.
- Compliance / audit: retain HarfangLab audit logs and authentication events alongside the rest of your security data.

## What do I need to use this integration?

You need:

- A running Elastic stack (`8.19+` or `9.1+`) — self-managed, Elastic Cloud, or Elastic Cloud Serverless.
- A HarfangLab Manager reachable from the host that runs Elastic Agent.
- For the **API** template: a static API token generated by an administrator from **Administration → Users → Generate token** in the HarfangLab Manager.
- For the **Syslog** template: the HarfangLab **Syslog connector** configured to forward events to the host running Elastic Agent over TCP or UDP.

## How do I deploy this integration?

### Agent-based deployment

Elastic Agent must be installed on a host that can reach (for the API) or be reached by (for syslog) the HarfangLab Manager. Follow the [installation instructions](https://www.elastic.co/guide/en/fleet/current/elastic-agent-installation.html). You can install only one Elastic Agent per host.

### Set up steps in HarfangLab

#### For the API policy template

1. Sign in to the HarfangLab Manager as an administrator.
2. Go to **Administration → Users**.
3. Click **Generate token** for a service account that has read access to alerts, threats, and endpoint inventory.
4. Copy the generated token; it will not be displayed again.
5. Note the full Manager URL including scheme and port (for example `https://tenant.example.com:8443`).

#### For the Syslog policy template

1. Sign in to the HarfangLab Manager as an administrator.
2. Go to **Administration → Connectors → Syslog**.
3. Add a new Syslog connector targeting the Elastic Agent host on TCP `9514` (or UDP `9515`).
4. Set the protocol to RFC 5424 with the JSON payload format.
5. Select which event types should be forwarded (alerts, threats, telemetry, etc.).

#### Vendor resources

- [HarfangLab documentation](https://docs.harfanglab.io/)
- [HarfangLab Manager API reference](https://docs.harfanglab.io/api/)
- [HarfangLab Syslog connector guide](https://docs.harfanglab.io/connectors/)

### Set up steps in Kibana

1. In Kibana, go to **Integrations** and search for **HarfangLab EDR**. Click **Add HarfangLab EDR**.
2. Choose a policy template:
   - **HarfangLab EDR (API)** — paste the Manager URL and the API token. Optionally adjust the per-stream polling interval (alerts, threats, agents) and the initial backfill window (`initial_interval`).
   - **HarfangLab EDR (Syslog)** — set the bind address (default `0.0.0.0`) and listen port (`9514` for TCP, `9515` for UDP). Enable TLS by populating the `ssl` block for TCP if your HarfangLab connector forwards over TLS.
3. Assign the integration to an agent policy that targets the host running Elastic Agent.
4. Save and deploy.

### Validation

1. In Kibana, open **Discover** and select one of the following data views:
   - `logs-harfang_labs.alert-*`
   - `logs-harfang_labs.threat-*`
   - `logs-harfang_labs.agent-*`
   - `logs-harfang_labs.event-*`
2. Confirm documents are flowing in. The HarfangLab Manager's UI counts should align with the indexed document counts over a comparable time window.
3. For the API template, the **Fleet** UI's per-agent logs will show CEL request activity (one HTTP request per polling round, with the persisted cursor).

## Troubleshooting

- No data on the API streams: verify the API token is valid (HarfangLab Manager → Administration → Users → token status). Confirm that the Manager URL is reachable from the agent host and that the agent's network can establish TLS to the configured port.
- 401 / 403 responses: regenerate the API token in HarfangLab and update the integration's `api_token` variable.
- API pagination loops or duplicate events: confirm the data stream's `initial_interval` is reasonable for your retention and is not set to a window longer than the API allows. Each stream tracks its own `last_seen` cursor.
- TCP syslog: no connection from HarfangLab: confirm the listener binds to `0.0.0.0` (default), the firewall on the agent host allows inbound TCP `9514`, and the HarfangLab Syslog connector uses framing `delimiter` with line delimiter `\n` (the default RFC 5424 over TCP framing). If TLS is enabled, the connector's CA must match the agent's `ssl.certificate_authorities`.
- UDP syslog: messages larger than `max_message_size` (default `10KiB`) are silently truncated. If HarfangLab emits oversized events (some thread / dump events contain large binary blobs), switch the connector to TCP.
- Pipeline errors visible in the `event.kind: pipeline_error` documents: search for `tags: preserve_original_event` to find affected events; the original payload is preserved in `event.original` for replay.

## Performance and scaling

For very high-volume HarfangLab deployments, prefer the syslog `tcp` input over `udp` so the kernel applies back-pressure and oversized messages are not truncated.

For very large endpoint inventories, increase the `agent` stream's polling `interval` (default `5m`) and reduce its `page_size` (default `100`) to reduce Manager API load.

The integration is stateless on the Elasticsearch side; standard data stream ILM, index lifecycle, and snapshot policies apply.

For more information on architectures that can be used for scaling this integration, check the [Ingest Architectures](https://www.elastic.co/docs/manage-data/ingest/ingest-reference-architectures) documentation.

### API usage

These HarfangLab REST API endpoints are polled by the integration:

- `GET /api/version` — health check on initial poll.
- `GET /api/data/alert/alert/Alert/` — alert listing with filters (`status`, `alert_type`, `level`, `alert_time__gte`, `ordering`).
- `GET /api/data/alert/alert/Threat/` — threat listing with filters (`status`, `level`, `creation_date__gte`, `ordering`).
- `GET /api/data/endpoint/Agent/` — agent inventory listing with filters (`status`, `lastseen__gte`, `ordering`).

All endpoints follow Django REST Framework pagination (`count`, `next`, `previous`, `results`).

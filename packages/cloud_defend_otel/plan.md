# Build `cloud_defend_otel` — OTel input package (PoC)

## Context

We are building a new Elastic package, `cloud_defend_otel`, that ports part of the
classic `cloud_defend` integration to the OpenTelemetry model. Instead of the
Beats-based `cloud_defend/control` input, this package is a **`type: input` OTel
input package** (`input: otelcol`) that configures the **`otlpreceiver`** in the
EDOT collector. Data arrives over OTLP and flows directly through the OTel pipeline
to Elasticsearch (the Elastic Agent injects the elasticsearch exporter — the
template defines only receivers/processors/pipelines, per
`docs/extend/otel-input-packages.md`).

A skeleton already exists at `packages/cloud_defend_otel/` (generated by
`elastic-package`, currently a classic `logfile` input skeleton). We will rewrite
it to follow the **`otlp_input_otel`** package, which is the closest existing
example (same receiver, sends OTLP straight to the agent, has a Go generator +
policy/system tests).

Related docs:
- @docs/extend/system-testing.md
- @docs/extend/otel-input-packages.md

### Decisions (from an interview)

- **Two datasets, realized via one template + a `data_stream.dataset` variable**
  (the `otlp_input_otel` pattern). Per-record attribute routing was investigated and
  is **not possible** in the OTel input-package model: Fleet auto-injects a
  `transform` processor that forces a single fixed `data_stream.dataset` per input
  instance and runs after any user transform. So the two datasets
  (`cloud_defend.logs`, `cloud_defend.file`) are realized as **two configured
  instances / two system-test configs** of the same template.
- **Dataset names:** `cloud_defend.logs` and `cloud_defend.file`.
- **Signal type:** logs only (`type: logs`). Both datasets are `type: log`.
- **Generator:** a Go generator emitting **generic OTLP logs** over gRPC to the
  agent's `otlpreceiver` (modeled on `packages/otlp_input_otel/_dev/deploy/docker/generator/main.go`).
- **No extra `policy_templates`:** a single `cloud_defendreceiver` policy template.

## Implementation steps

### 1. `manifest.yml` (rewrite) — DONE
Model on `packages/otlp_input_otel/manifest.yml`, but logs-only and cloud_defend-branded.

- `format_version: 3.6.0`, `name: cloud_defend_otel`, `type: input`.
- `title`/`description`: cloud_defend OTLP wording (BETA, keep cloud-defend phrasing).
- `categories: [security, containers, kubernetes, observability, opentelemetry]`
  (always include `opentelemetry`).
- `conditions.kibana.version: "^9.2.0"` (logs-only → `^9.2.0` per the docs version
  table). `conditions.elastic.subscription: "enterprise"` +
  `capabilities: [security]` (preserve cloud_defend's licensing).
- `icons:` point at `/img/cloud_defend_otellogo.svg`. Drop `screenshots:`.
- Drop the skeleton's `elasticsearch.index_template.mappings.subobjects: false`
  (not used by `otlp_input_otel`).
- Single policy template:
  ```yaml
  policy_templates:
    - name: cloud_defendreceiver
      type: logs
      title: Defend for Containers (OpenTelemetry)
      description: Receive cloud_defend telemetry via OTLP and route to a dataset.
      input: otelcol
      template_path: input.yml.hbs
      vars:
        - name: grpc_endpoint
          type: text
          required: true
          show_user: true
          default: localhost:4317
        - name: http_endpoint
          type: text
          required: true
          show_user: true
          default: localhost:4318
        - name: data_stream.dataset
          type: text
          required: false
          show_user: true
          title: Data Stream Dataset
          description: Dataset to write data to (e.g. cloud_defend.logs or cloud_defend.file).
          default: cloud_defend.logs
  ```
- `owner: {github: elastic/cloud-services, type: elastic}` (keep from skeleton).

### 2. `agent/input/input.yml.hbs` (rewrite) — DONE
Minimal `otlpreceiver` template (logs-only) based on `otlp_input_otel` but stripped
of auth/TLS for the PoC. Must define **only** receivers/processors/pipelines —
no exporters (agent injects them). Must include `resourcedetection` (submission
requirement).
```handlebars
receivers:
  otlp:
    protocols:
      grpc:
        endpoint: {{grpc_endpoint}}
      http:
        endpoint: {{http_endpoint}}
processors:
  resourcedetection/system:
    detectors: ["system"]
service:
  pipelines:
    logs:
      receivers: [otlp]
      processors: [resourcedetection/system]
```

### 3. `docs/README.md` + `_dev/build/docs/README.md` — DONE
Write `_dev/build/docs/README.md` with `{{- generatedHeader }}` header (see
`packages/kafka_input_otel/_dev/build/docs/README.md`). Sections: Overview
(uses the [otlpreceiver](https://github.com/open-telemetry/opentelemetry-collector-contrib/tree/main/receiver/otlpreceiver)),
How it works (OTLP → EDOT collector → agent → Elasticsearch), Configuration
(endpoints + `data_stream.dataset`, note the two datasets), link to upstream docs.
`docs/README.md` is rendered by `elastic-package build`.

### 4. Policy tests (`_dev/test/policy/`)
Model on `packages/otlp_input_otel/_dev/test/policy/`:
- `test-default.yml`: `vars: ~` → default dataset `cloud_defend.logs`.
- `test-file-dataset.yml`: `vars: {data_stream.dataset: cloud_defend.file}`.
- Generate `.expected` files with `elastic-package test policy --generate` (do not
  hand-write). Verify the injected `transform/componentid-1` sets
  `data_stream.dataset` to the configured value and `data_stream.type` to `logs`.

### 5. System tests + docker (`_dev/test/system/`, `_dev/deploy/docker/`)
Model on `packages/otlp_input_otel/_dev/deploy/docker/` (generator-only compose;
generator sends OTLP straight to `elastic-agent:4317`, waits for `SIGHUP`).
- `docker-compose.yml`: single `generator` service (`build: .`), no intermediate
  otelcol (the agent IS the receiver). gRPC only for the PoC.
- `Dockerfile`: golang builder → alpine, same as `otlp_input_otel`.
- `generator/main.go`: **logs-only**, gRPC-only — trim the `otlp_input_otel`
  generator to just `sendLogs()` targeting `elastic-agent:4317`, keep the
  SIGHUP-wait. `go.mod`/`go.sum`: trim to the otlplog/grpc deps (drop metric/trace
  exporters).
- `_dev/test/config.yml`: `system: {parallel: false}`.
- System configs (both set `service: cloud_defend_otel`,
  `service_notify_signal: SIGHUP`, `grpc_endpoint: 0.0.0.0:4317`,
  `http_endpoint: 0.0.0.0:4318`, `assert.min_count`):
  - `test-logs-config.yml`: `data_stream.dataset: cloud_defend.logs`.
  - `test-file-config.yml`: `data_stream.dataset: cloud_defend.file`.
  Both reuse the same generator; the dataset differs only by the agent var.

3. `elastic-package stack up -d` then `elastic-package test system` — generator
   pushes OTLP logs through the agent's `otlpreceiver` to Elasticsearch; both
   system configs meet `assert.min_count` in `logs-cloud_defend.logs-*` and
   `logs-cloud_defend.file-*`.
4. `elastic-package test system --generate` to produce `sample_event.json`.
5. Confirm submission checklist (`docs/extend/otel-input-packages.md`): `type:
   input` + `input: otelcol`, `opentelemetry` category, `resourcedetection`

   CODEOWNERS covers the path.
```

## Open risk

The agent-injected `transform` (verified via `.expected` outputs) is what makes the
single-dataset-per-instance behavior; the two-instance approach sidesteps it
cleanly. If a future requirement needs true per-record routing into both datasets
from one stream, that is **not** supported by the current input-package model and
would need a different (non-input-package) approach.
